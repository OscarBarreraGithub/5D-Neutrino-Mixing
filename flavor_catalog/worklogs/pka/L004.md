# PKA Worklog: L004 mu-e Conversion in Gold

I read the required plan, orchestrator decisions, Opus signoff, process
template, and catalog README before creating files.  The assigned row is
charged-lepton process L004, mu-e conversion in gold, with initial code
coverage `NOT-SEEN` and the SINDRUM II 2006 bound as the expected central
source.

Searches and source checks focused on `PDG SINDRUM II muonic gold 7e-13`,
`A search for mu-e conversion in muonic gold DOI`, `Bertl 2006 SINDRUM II
gold Crossref`, `Mu2e Technical Design Report 1501.05241`, `COMET Phase-I
Technical Design Report 1812.09018`, and `arXiv 0804.1954 RS flavor`.  I used
the streamed PDG live muon listing for the canonical value and extracted only
the relevant text block; the PDF was not stored.  I used Crossref metadata for
the SINDRUM II paper identity and avoided the Springer full text/PDF.  JuSER
and DORA pages were checked, but JuSER returned a web challenge and DORA served
a JavaScript shell in this environment, so I did not use either as a tracked
L004 source.  ResearchGate and publisher PDF copies were rejected.

The value recorded is
`sigma(mu- Au -> e- Au) / sigma(mu- Au -> capture) < 7e-13` at `90% C.L.`,
with PDG document id `BERTL 06` and comment `SINDRUM II`.  Post-2008 context
is limited intentionally: no new gold result supersedes the PDG line, while
Mu2e and COMET aluminum programs set the near-term comparison scale.

For coverage I ran the three required catalog grep commands over
`quarkConstraints`, `qcd`, `flavorConstraints`, `neutrinos`, `yukawa`,
`warpConfig`, `solvers`, `scanParams`, and `tests`, plus a targeted
SINDRUM/Mu2e/COMET/R_mue/gold-conversion query.  The targeted query returned
no hits.  Generic `conversion` hits were unrelated unit or bag-parameter
conversion internals.  The only adjacent implementation is the dipole-only
`mu->e gamma` checker in `flavorConstraints/muToEGamma.py` and its scan call
in `scanParams/scan.py`.
