## Scope

Three of the four issues closed here are the same documented seam: the V/P
counts written into the `flavor-catalog-v0.{3,4}` git tag annotations
diverge from the canonical post-consolidation totals carried by the
master-compile reports (R18-I1, R19-I3) and that drift leaked into one
operator-facing doc (R21-I1: `WEBSITE_RUNBOOK.md:21`). C07 reconciled the
master-compile reports themselves; C12 reconciles the *cross-document*
seam — the tags are frozen, so the canonical numbers are pinned in
`flavor_catalog/audits/factcheck_status.md` and downstream docs are
aligned to that single source of truth.

R22-I3 is a structurally adjacent UX cosmetic on the website (dead
`data-tier` wiring on `<tr>` rows in `EntryTable.astro` after the phase-10
tier-badge removal at `7053fb7`). It is closed by the same C12 commit
because it is a one-line cleanup with no semantic impact.

This is docs+cosmetic only. No physics, no code that affects tests, no
master-compile bump.

## Why option (c) — not (a) tag-rewrite, not (b) corrected-tags

Per the dispatch prompt:

- Option (a) — modifying tag annotations — would rewrite tag SHAs and
  diverge from origin. The v0.3 / v0.4 tags are public and already
  referenced from `master_compile_v0{3,4}_report.md`, the catalog
  README, and the website. Rewriting history for a cosmetic count seam
  is the wrong trade.
- Option (b) — adding `flavor-catalog-v0.X-corrected` tags — proliferates
  tags for what is fundamentally a single documentation seam. It also
  introduces a new ambiguity (which tag is canonical?) where there was
  only one before.
- Option (c) — documenting the drift in a single canonical provenance
  block (now in `factcheck_status.md`) and aligning the one downstream
  doc that propagated the stale number (the website runbook) — is the
  minimum-surface fix that produces a single source of truth without
  rewriting any history.
