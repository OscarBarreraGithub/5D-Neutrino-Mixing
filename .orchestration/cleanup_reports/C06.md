# C06 — embed `git_sha` in `tile_summary.json`

**Tier:** 3 · **Lane:** opus · **Issues closed:** R07-I3 INFO · **Status:** DONE

## Scope

R07-I3 (review `.orchestration/reviews/R07.md`, ISSUES.md:84-88) flagged
that the canonical run-time SHA in `artifacts/quarkscan_paper_rc1_manifest.json`
(field `code_sha_at_run_time = 29803ff…`) had to be inferred from UTC
directory stamps because `tile_summary.json` did not embed a git SHA.
The inference logic is transparent and reproducible, but a future
re-runner cannot programmatically verify that the canonical SHA actually
produced the on-disk outputs. The recommended fix was forward-looking:
write `git rev-parse HEAD` into `tile_summary.json` alongside the
existing metadata.

This unit lands that forward-looking change. The fix is purely
provenance-additive — no physics, no numerics — and is fully backward
compatible: old `tile_summary.json` files (with no `git_sha` key) still
load, and downstream consumers can default the missing field to
`"unknown"` exactly as the manifest's `code_sha_at_run_time_basis`
prose already documents for historical scans.

## Implementation

1. **Writer (`scripts/run_rs_anarchy.py`)** — the sole producer of
   `tile_summary.json` in the repo (`grep -rn 'tile_summary'` over
   `scripts/ scanParams/ quarkConstraints/` returns this file only as
   a writer; all other hits are readers/plotters).

   - Added `import subprocess` to the module top-of-file imports.
   - Added a module-level helper `_resolve_git_sha()` (immediately
     after `_load_pdg_targets`, before the "Anarchic Y sampler"
     section) that calls
     `subprocess.check_output(['git', 'rev-parse', 'HEAD'],
     cwd=str(_REPO_ROOT), stderr=subprocess.DEVNULL)`,
     `.decode().strip()`. Wrapped in a `try/except` over
     `(CalledProcessError, FileNotFoundError, OSError)` that returns
     the documented `"unknown"` sentinel — matching the manifest's
     existing inference-fallback convention. Empty output also maps
     to `"unknown"`.
   - Added `"git_sha": _resolve_git_sha()` to `summary_payload`
     (top-level, alongside `config`, `elapsed_seconds`, `tiles`,
     `schema`, `convention`). Placed at the top level rather than
     under `config` because it describes the *executing* code state,
     not a configured input.

2. **Loaders / consumers** — no changes required. The four
   `tile_summary.json` readers
   (`scripts/plot_rs_anarchy_summary.py`,
   `scripts/rs_anarchy_cfw_comparison.py`,
   `scripts/rs_anarchy_gate_sensitivity.py`,
   `scripts/rs_anarchy_mkk_min_hist_by_cvals.py`) all do plain
   `json.load`/`json.loads` and pull specific keys (`config`,
   `tiles`, `convention`). Adding a new top-level field is invisible
   to them. Old files without the field continue to load via the
   same `json.load` path; consumers that want to verify against a
   manifest can use `payload.get("git_sha", "unknown")`.

3. **Tests (`tests/test_run_rs_anarchy.py`)** — appended a new
   block of four tests under the existing `_test_run_rs_anarchy_module`
   fixture:

   - `test_resolve_git_sha_inside_repo_returns_hex` — calls
     `_resolve_git_sha()` from this checkout and asserts the return
     value matches `^[0-9a-f]{40}$` or equals the `"unknown"`
     fallback (defensive against runners without `git` on PATH).
   - `test_resolve_git_sha_outside_git_tree_returns_unknown` —
     `monkeypatch`es `rs_module._REPO_ROOT` to a `tmp_path` and
     asserts the helper returns `"unknown"` without raising.
   - `test_tile_summary_embeds_git_sha` — drives
     `rs_module.main(["--output-dir", tmp, "--n-draws", "1",
     "--m-kk-tev", "10", "--n-workers", "1", "--base-seed",
     "20260525"])`, asserts the script exits `0`, reads the produced
     `tile_summary.json`, and asserts `"git_sha"` is present, is a
     non-empty string, and matches the hex-or-`"unknown"` shape.
   - `test_legacy_tile_summary_without_git_sha_still_loads` —
     constructs a minimal pre-C06 payload (no `git_sha` key) and
     confirms `json.loads` succeeds, `"git_sha" not in loaded`, and
     `loaded.get("git_sha", "unknown") == "unknown"`. This pins the
     backward-compat contract directly.

1. **New writes contain `git_sha`.** Confirmed by
   `test_tile_summary_embeds_git_sha` — the script's full
   `main()` is invoked with `n_draws=1, m_kk=10 TeV, n_workers=1`,
   and the on-disk file is parsed back. Field present, hex-shaped.
2. **Broader test selection.**
   `timeout 600 python -m pytest tests/ -k 'tile or summary or scan or dispatch' --tb=short`
   → `55 passed, 515 deselected in 122.46s`. The selector matches
   the dispatch-style and scan tests for both the legacy and modern
   lanes; targeted run of the four new tests
   (`pytest tests/test_run_rs_anarchy.py -k 'git_sha or legacy_tile_summary'`)
   → `4 passed in 5.42s`.
3. **Old `tile_summary.json` files still load.** Asserted directly
   by `test_legacy_tile_summary_without_git_sha_still_loads`, and
   cross-checked structurally: all four current readers use
   `json.load(open(path))` / `json.loads(path.read_text())` and
   index specific keys (`config`, `tiles`, `convention`) — none
   require `git_sha`, none validate against a schema.
