## 2. R07-I2 — k>0 Wilson upper-limit test coverage

### 2.1 Change

Added three new test groups to `tests/test_finite_stats.py` (file went from 21 → 99 lines):

1. `test_wilson_upper_limit_k_gt_zero_matches_closed_form` — **parametrized** over `(k, n) ∈ {(1, 100), (1, 1000), (5, 1000), (10, 1000), (50, 1000)}` (5 cases). Cross-checks `quarkConstraints.finite_stats.wilson_upper_limit(k, n, z=1.92)` against an in-file closed-form helper `_wilson_upper_limit_closed_form(k, n, z)` that implements the standard Brown–Cai–DasGupta (Stat. Sci. 2001) formula
   ```
   UL = (p_hat + z^2/(2n) + z * sqrt(p_hat*(1-p_hat)/n + z^2/(4n^2))) / (1 + z^2/n)
   ```
   at the repo's z=1.92 convention. Tolerance `rel=1e-12, abs=1e-15`. Both formulations agree to floating-point.

2. `test_wilson_upper_limit_k_gt_zero_matches_scipy_at_z_eq_1p96` — **parametrized** over `(k, n) ∈ {(1, 100), (5, 1000), (50, 1000)}` (3 cases). Uses `scipy.stats.binomtest(k=k, n=n).proportion_ci(method='wilson', confidence_level=0.95).high` as oracle and feeds `z = scipy.stats.norm.ppf(0.975) = 1.959964...` into the repo helper. Guarded by `pytest.importorskip("scipy", minversion="1.10")`. Tolerance `rel=1e-12, abs=1e-15`. Passes.

3. `test_wilson_upper_limit_k_gt_zero_spot_check_k5_n1000` — explicit numerical pin at `k=5, n=1000, z=1.92` to value `~0.01146`. **Note**: the C04 dispatch prompt's Check-3 hand-computation said "around 0.0089". That value is incorrect — it omits the `+z * sqrt(p_hat*(1-p_hat)/n + z^2/(4n^2))` margin term. The actual closed-form evaluates to `0.011463261666780364` (computed twice — once by the in-file helper, once by the repo's `wilson_upper_limit`; both agree). The test docstring carries an explicit note flagging the dispatch-prompt error so future readers don't propagate it.

### 2.2 Plan choice

The plan §C C04 left the choice between scipy oracle and closed-form fallback open ("either (a) or (b)"). C04 **uses both**: the closed-form group is self-contained and survives any future scipy version movement; the scipy group is the requested oracle cross-check and confirms the repo helper matches the standard 95% Wilson interval at the corresponding z. Together they pin the k>0 behavior of the helper from two independent directions.

## 3. R02-I2 — disposition (deferred to C13)

The dispatch prompt mentioned R02-I2 and asked whether the spurion-seed test ("`test_quark_fit.py`") "only covers k=0". The honest answer is: **there is no "k" parameter here**. The spurion-seed test suite exercises `default_spurion_seed()` (the deterministic LSQ-converged benchmark seed defined in `quarkConstraints/benchmarks.py:189-223`) along multiple orbits in the canonical-vector quotient: the default seed itself, `_quotient_equivalent_seed(default_spurion_seed())` which shifts left/right rotation angles by multiples of 2π and rescales `overall_scale`, and several ad-hoc seeds in `test_canonical_vector_quotients_out_*` with non-trivial right-rotation angles. This is the analog of "k > 0" coverage in the canonical-vector sense — the suite is not stuck at one trivial representative.

C04 added a module-level docstring block at the top of `tests/test_quark_fit.py` (~25 lines) recording this analysis explicitly so a future reader cannot mistake the existing coverage for a "k=0 only" gap. The block also cross-references C13 for the underlying R02-I2 fix: the **in-source provenance comment on the literal arrays inside `default_spurion_seed()`** (e.g. "values are the LSQ-converged optimum from the previous seed under `default_quark_targets()` at mu = 163.5 GeV; overall_scale = 2.8 preserved"). That literal-comment edit belongs in `quarkConstraints/benchmarks.py` and is the actual R02-I2 action item; C13 owns it per CLEANUP_PLAN.md §C.

### Check 3 — physics correctness of the k>0 Wilson UL?

Hand-verified at `k=5, n=1000, z=1.92`:
- `p_hat = 0.005`
- `z^2 = 3.6864`, `z^2/(2n) = 1.8432e-3`, `z^2/(4n^2) = 9.216e-7`
- `p_hat*(1-p_hat)/n = 4.975e-6`
- `sqrt(4.975e-6 + 9.216e-7) = sqrt(5.897e-6) = 2.4283e-3`
- `denom = 1 + z^2/n = 1.0036864`
- `center = (0.005 + 1.8432e-3) / 1.0036864 = 6.8181e-3`
- `margin = 1.92 * 2.4283e-3 / 1.0036864 = 4.6453e-3`
- `UL = center + margin = 1.14634e-2` ≈ **0.01146** ✓

This matches the repo helper bit-for-bit (`0.011463261666780366`) and the in-file closed-form (`0.011463261666780364`). The dispatch-prompt suggested ~0.0089 is incorrect (margin term omitted); the test docstring carries an explicit note.
